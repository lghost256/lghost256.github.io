Convierte Linux en un conmutador (switch) de software con NetworkManager

19 junio 2025
  • Hacer persistente la configuración del bridge

Algo de contexto

Hace unas semanas, empecé a elaborar un laboratorio en GNS3 para repasar mis habilidades en detección de intrusiones. Instalé Wazuh en un servidor, los agentes en los endpoints, y todo funcionó correctamente. Pero enseguida me di cuenta de que me faltaba una parte clave en mi detección: datos de red. Decidí usar Suricata para recoger esos datos, pero me topé con un problema: los switches virtuales en GNS3 son útiles para reenviar tramas de nivel 2, pero no cuentan con funciones avanzadas como el port mirroring (que permitiría enviar copias del tráfico a Suricata para monitorizarlo). Sin esta función, tuve que buscar una alternativa para que Suricata pudiera capturar todo el tráfico de mi red.

Descarté utilizar un hub. Entonces se me ocurrió una idea: ¿y si colocara el servidor Suricata en medio y lo utilizara como un switch? Todo el tráfico pasaría por ahí, por lo que sería sencillo de monitorizar. Aunque esta no es una guía de configuración de Suricata, sí te mostraré cómo convertí ese servidor en un switch software. El proceso es sencillo y funciona bien en un entorno de laboratorio como el que describo.

Pasos de configuración

NOTA: Estoy usando Rocky Linux 8 en este laboratorio, que trae NetworkManager preinstalado. Todos los pasos aquí sirven para sistemas que lo usan.

Esta es la topología con la que voy a trabajar:

Como ves, hay dos hosts (PC1 y PC2) y una conexión a Internet. Como en GNS3 puedo añadir tantas tarjetas de red como quiera con un clic, añadí 9 adaptadores a Suricata. Para identificar las interfaces activas, usé:

ip link show | grep "state UP"

Ahora, nos conectamos a Suricata y creamos una nueva interfaz bridge (en este caso, br0):

nmcli con add type bridge ifname br0 con-name br0

Las interfaces bridge en Linux sirven para:

  • Reenviar tramas Ethernet entre interfaces miembros (basado en aprendizaje de MAC).
  • Aprender qué MACs están en qué puertos (interfaces).
  • Descartar o inundar destinos desconocidos o inalcanzables.
  • Opcionalmente soportan:
    • Spanning Tree Protocol (STP)
    • Filtrado VLAN

Como no estoy trabajando con VLANs ni STP en este laboratorio, sólo voy a mostrar cómo añadir interfaces como miembros del bridge para crear un dominio de broadcast. Ten en cuenta que en entornos VLAN-aware, los bridges Linux también soportan etiquetado y filtrado VLAN vía bridge vlan y ip link.

Por ahora, vamos a añadir las interfaces de la topología como miembros de br0. Pero cuidado: ninguna interfaz esclava puede tener configuración IP. Si no, fallará. En mi caso, borré todas las conexiones configuradas por defecto con:

nmcli con show
nmcli con delete <nombre-de-la-conexión>

Después, añadimos las interfaces al bridge. Es recomendable usar con-name para mantenerlo organizado, aunque es algo opcional:

nmcli con add type ethernet ifname ens3 master br0 con-name ens3-slave
nmcli con add type ethernet ifname ens4 master br0 con-name ens4-slave 
nmcli con add type ethernet ifname ens10 master br0 con-name ens10-slave

Cuando esté listo, levantamos la conexión:

nmcli con up br0

Por último, para que Suricata se comunique en la red IP, ¿a qué interfaz asignamos la IP? La respuesta es: a br0. En mi caso, con la red 10.0.10.0/24 usé:

nmcli con mod "br0" \
ipv4.addresses "10.0.10.50/24" \
ipv4.gateway "10.0.10.1" \
ipv4.dns "8.8.8.8" \
ipv4.method "manual"

Recuerda que sólo el bridge (br0) debe tener IP; las interfaces esclavas (ens3, ens4, etc.) no tienen configuración de capa 3.

Tras estos pasos, la configuración del bridge está casi lista… Pero ojalá pudiera dejar todo configurado de manera explícita en ficheros para no tener que repetir comandos… Un segundo, ¡sí que puedo hacerlo!

Hacer persistente la configuración del bridge

Para que los cambios sobrevivan a reinicios, hay que definirlos en /etc/NetworkManager/system-connections/, donde están los ficheros .nmconnection.

Primero, creo el archivo br0.nmconnection con la configuración del bridge:

[connection]
id=br0
type=bridge
interface-name=br0
autoconnect=true

[bridge]
stp=false

[ipv4]
method=manual
address1=10.0.10.50/24
gateway=10.0.10.1
dns=10.0.10.1;8.8.8.8;

[ipv6]
method=ignore

Luego, para cada interfaz esclava, creo su propio archivo, por ejemplo ens3.nmconnection:

[connection]
id=ens3-slave
type=ethernet
interface-name=ens3
master=br0
slave-type=bridge
autoconnect=true

Después, establezco permisos -rw------- para que NetworkManager los tome en cuenta. Si no lo hago, NetworkManager ignorará estas configuraciones:

chmod 600 /etc/NetworkManager/system-connections/*.nmconnection

Finalmente, recargo la configuración:

nmcli connection reload

Con estos pasos, Suricata puede ahora funcionar como switch y monitorizar todo el tráfico de mi red.
